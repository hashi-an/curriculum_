IT用語まとめ

プルリクエスト（マージリクエスト）

GitHubの機能：開発者のローカルリポジトリでの変更をレビュワーに通知し、マージを依頼する機能。

  • 機能追加や改修など、作業内容をレビュー・マージ担当者やその他関係者に通知する。
  • ソースコードの変更箇所をわかりやすく表示できる。
  • ソースコードに関するコミュニケーションの場を提供。

Git Flow

Git をチームで使う上での運用方法を表したブランチモデルをサポートするツール（コマンド）。GitFlow は複雑なので、GitHubFlowなどシンプルなフローを使われることが多い。

  • git-flowは2種類のメインブランチ ( master , develop ) と3種類のサポートブランチ ( feature , release , hotfix )を切替ながら開発が進められる。
  • git-flwo詳細　参照ページ
  • GitHub Flowではメインのmasterブランチと機能開発のためのfeatureブランチの2つのみのシンプルな構成になっている。
  • git-flowにはない特徴としては、Pull Requestを使うという点と、masterブランチは常にデプロイ可能という点がある。

CRON（クーロン）

Linuxを運用する上で欠かせない定期的にタスクを自動実行するためのツール。

  • GitHub の自動化ツール GitHub Actions のワークフローを使って特定の処理を定期的に実行する際に利用できる。
  • default branch以外は、cronの対象にならない、5分以下は動かない。
  • 設定時間から5分程のズレが生じる。

    【RDBMS】：ベンダー製品のOracle Db（Oracle）、SQL Server（Microsoft）、DB2（IBM）やオープンソースのMySQL、そこから派生したMaria DB、Postgre SQLなどがある。

    【CRUD】：Create（登録）、Read（参照）、Update（更新）、Delete（削除）機能をまとめた表現。

    【トランザクション】：複数の SQL 文によるデータ更新を1つの処理としてまとめてデータベースに反映させること。
    　　[コミット]：一連の処理が問題なく、成功すれば整合性を保って更新される。こうして、結果が確定されることを指す。
    　　[ロールバック]：一連の処理に問題が生じてしまい、更新するはずだった状態から開始直前の状態に戻すことを指す。

    【排他制御（排他ロック）】：複数のアクセスが見込まれる場合に、同時アクセスにより不整合が発生することを防ぐため、他トランザクションからはアクセスできないようにして直列に処理されるように制御すること。
    　　[楽観ロック]：「同時アクセスはあまり起きない」という楽観的な考え方の排他制御。データ自体をロックすることはせず、更新対象のデータが更新前と同じであるかを確認してから更新することで、データの整合性を保つ排他制御。
    　　[悲観ロック]：同じデータを他者が頻繁に更新するという悲観的な考え方の排他制御。更新対象データを取得時にロックして他者が操作できないようにする方式。

    【実行計画】：SQLserverがユーザーが発行したSQLを解析し決定した実データに対して、どのようにアクセス・計算を行うかが記載されている計画書。SQLはDBの「どの」データを取得するかをユーザーが指定するが、「どのように」そのデータを探して取得するかはSQLserverのオプティマイザ機構が決定する。その手順内容が示されたもの。

    【チューニング】：データベースの処理・データベースの論理設計・SQLを調整することでパフォーマンスを向上させる目的で行う。 →参考ページ1

Gitについて

  • Gitとは、ソースコードなどの変更履歴を管理する分散型バージョン管理システム。 実際の開発現場では、複数人が同じファイルを編集することがある。
  • リポジトリ
    チーム内で同じファイルを同時に編集してしまい、どちらが最新のファイルなのかわからなくなる。という事態を防ぐこともできる。 ソースコードの変更履歴を保管する場所のことを「リポジトリ」という。 Gitでは、ネットワーク上にこの「リポジトリ」と呼ばれる場所を作り、そこにファイルをアップロードするような感覚で使う。
  • リモートリポジトリ
    ネットワーク上にあるのでURLを知っている人は、誰でもその保管場所（リポジトリ）を見にいくことができる。 みんなで共有するために、ネットワーク（リモート）上にある保管場所のことを「リモートリポジトリ」という。
  • ローカルリポジトリ
    リモートリポジトリに置いてあるファイルを複製して自分のPC（ローカル）に取り込むことができる。 デスクトップにフォルダを作って保存するといったように、普段自分たちが開発するために使っているPCにもファイルを保存する場所がある。 そこを「ローカルリポジトリ」として設定して「リモートリポジトリ」と繋げることでファイルを管理していく。

    • 「リモートリポジトリ」：ネットワーク上でみんなが共有しているフォルダ
    「ローカルリポジトリ」：個人のPCで管理しているフォルダ
    最新版を「リモートリポジトリ」にアップロードすると、他の人もその最新版を「リモートリポジトリ」からダウンロードできるようなイメージ。


    ◆Gitでよく使う用語

  • クローン
    自分のローカル環境にリモートリポジトリの複製(ローカルリポジトリ)を作ること。 プロジェクトにアサインしたら、環境構築の時にほぼ間違いなく通る。
  • コミット
    自分の作業したファイルの変更履歴をローカルリポジトリに保存すること。
  • プッシュ
    ローカルリポジトリの変更履歴をリモートリポジトリに反映すること。 これによって、ネットワーク上のリモートリポジトリから他の人が自分の変更履歴を確認できるようになる。
  • マージ
    ブランチの変更履歴の情報を他のブランチに流し込むこと。 マージされたブランチはマージ元のブランチの変更履歴をすべて持っていることになる。
  • プル
    リモートリポジトリにあるブランチの変更履歴の情報をローカルリポジトリのブランチに流し込むこと。 動作としてが、リモートリポジトリに更新があるか確認(フェッチ)して、更新を取り込む(マージ)になるため、フェッチ + マージ = プルということになる。
  • コンフリクト
    変更履歴の競合のこと。 AさんとBさんで同じファイルで作業したりして、１つのファイルに対して別々の変更履歴があると発生する。
